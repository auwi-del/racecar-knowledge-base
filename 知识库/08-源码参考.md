## 1. racecar_driver：底盘驱动 (`racecar_driver_node.cpp`)

**包路径**：`~/racecar/src/racecar_driver/`

### 关键类：`RacecarDriver` (继承 `rclcpp::Node`)

```cpp
// 构造函数
RacecarDriver() {
    // 声明参数：serial_port=/dev/car, baud_rate=38400
    // 初始化串口：art_racecar_init(baud_rate, serial_port)
    
    // 订阅 /car_cmd_vel → TwistCallback（导航/程序控制）
    // 订阅 /teleop_cmd_vel → teleop_TwistCallback（遥控/循迹）
}
```

### 回调函数

**TwistCallback** (处理 `/car_cmd_vel`)：
```cpp
linear_x = twist->linear.x * 100 + 1500;
angle = 1500.0 - twist->angular.z * 1300;
send_cmd(linear_x, angle);
```
- 线速度被缩放为 `linear.x * 100 + 1500`（映射到 PWM 范围 1500±100）
- 角速度被缩放为 `1500 - angular.z * 1300`（负号：正角速度=左转, PWM 值减小=左转）

**teleop_TwistCallback** (处理 `/teleop_cmd_vel`)：
```cpp
angle = 2500.0 - twist->angular.z * 2000.0 / 180.0;
linear_x = twist->linear.x;   // 直接使用原始值
send_cmd(linear_x, angle);
```
- 线速度直接透传（由上游脚本计算好）
- 角速度从度→PWM 的转换

### 底层接口：`racecar_driver.h`

```cpp
int Open_Serial_Dev(char *dev);                    // 打开串口设备
int art_racecar_init(int speed, char *dev);        // 初始化（波特率, 串口路径）
unsigned char send_cmd(uint16_t motor_pwm, uint16_t servo_pwm);  // 发送电机和舵机 PWM 值
```

注意：`racecar_driver_node` 和 `racecar_driver_node_one` 的唯一区别是订阅的 Twist 话题不同：
- `racecar_driver_node` → 订阅 `/car_cmd_vel`（默认）和 `/teleop_cmd_vel`
- `racecar_driver_node_one` → 订阅 `/cmd_vel`（默认）和 `/teleop_cmd_vel`

## 2. encoder：编码器节点 (`encoder_node.py`)

**包路径**：`~/racecar/src/encoder/`

### 串口通信协议

发送查询命令 → 接收响应 → 解析速度/方向：

**读速度**：
```
发送: 01 03 00 03 00 01 74 0A
响应: 01 03 02 [high_byte] [low_byte] [checksum]
速度值 = (high * 256 + low)  （若 >= 32768 则减 65536）
```

**读方向（圈数）**：
```
发送: 01 03 00 00 00 01 84 0A
响应: 01 03 02 [high_byte] [low_byte] [checksum]
方向 = 当前圈数 - 上次圈数 → 判断正负
```

### 速度计算公式

```python
C = 227.45 / 1000    # 编码器常数（mm/脉冲）
Vel = 0.714 * angle_v * C / 1024.0 / 0.02 * k * 0.25 * 1.03
```

其中：
- `0.714` = 角度→速度的转换系数
- `C` = 227.45 / 1000 = 每脉冲移动距离（米）
- `1024` = 编码器分辨率（每圈脉冲数）
- `0.02` = 定时器周期（50Hz）
- `k` = 可调系数（来自参数，默认1）
- `0.25` = 减速比相关
- `1.03` = 修正系数

### Odometry 消息

```python
pub_vel = Odometry()
pub_vel.header.frame_id = 'odom'           # 父坐标系
pub_vel.child_frame_id = 'base_footprint'  # 子坐标系
pub_vel.twist.twist.linear.x = Vel         # X 方向线速度
```

注意：编码器发布的 Odometry 的 `frame_id='odom'`，而 EKF 融合后的里程计使用 `odom_combined`。

## 3. lidar_tracking：激光雷达锥桶循迹 (`lidar_tracking.cpp`)

**包路径**：`~/racecar/src/lidar_tracking/`

### 核心类：`PubAndSub` (继承 `rclcpp::Node`)

#### 主流程 (`callback`)

```cpp
void callback(const sensor_msgs::msg::LaserScan::SharedPtr laser) {
    updateDrivingMode(laser);           // 判断有无锥桶 → 切换模式
    if (is_no_cone_mode) {
        handleNoConeMode();             // 无锥桶巡航
    } else {
        handleNormalTrackingMode(laser); // 正常循迹
    }
    // 角度限幅 [0, 180]
    pub_->publish(twist);
}
```

#### 步骤1：锥桶检测 (`detectCones`)

检测条件（距离突变+平台稳定性）：
1. 相邻扫描点距离突变更大的点：`ranges[i-1] - ranges[i] >= 1.5`
2. 该点距离 < 2.5m  
3. 前后一致性：`ranges[i-1] - ranges[i+1] >= 1.5`
4. 后续最多 10 个点中，至少 6 个在该点 0.2m 以内（形成"平台"）

分类规则：
- `tmp_x > 0` → **红锥桶**（右侧）
- `tmp_x < 0` → **蓝锥桶**（左侧）

窗口过滤：`x ∈ [-max_right_dis, max_left_dis]`, `y ∈ (-0.2, 2)`

#### 步骤2：数据关联 (`associateCones`)

处理锥桶序列中的断裂。当一侧锥桶数量显著多于另一侧时，查找距离突变点（相邻距离 > 1.4m），将断裂后的锥桶"移交"到另一侧。

这样做的原因是：赛道中锥桶交替排列，某一侧后段可能实际上是另一侧的前段（扫描起始位置不同导致）。

#### 步骤3：标准化 (`standardizeCones`)

- 每侧最多保留 3 个点
- 通过复制近点使红蓝数量一致（1→2→3 补齐）
- 最终红蓝锥桶数量相等，最多 3 对

#### 步骤4：误差计算与 PID

```cpp
// 加权平均误差
error = sum( (blue[i].x + red[back-i].x) * rate_p[i] ) / sum(rate_p) * 53.33;

// PID
angle = pid.PIDPositional(error);

// 最终输出
twist.angular.z = 75.5 + angle;   // 基础舵角 75.5
twist.linear.x = speed;            // 基础速度
if (abs(angle) >= 20) twist.linear.x -= 21;  // 大角度减速
```

#### 无锥桶巡航 (`handleNoConeMode`)

当扇形检测区域内（范围 1.3m，角度 ±84°）无物体时，进入无锥桶巡航：
- 低速左转搜索：`twist.angular.z = no_cone_turn_angle`（约 105°）
- `twist.linear.x = no_cone_speed`（约 1520）

## 4. waypoint_cycle：航点循环 (`waypoint_cycle.py`)

**包路径**：`~/racecar/src/nav2_waypoint_cycle/`

### 核心类：`waypoint_cycle` (继承 `Node`)

**工作流程**：
1. 用户在 RViz2 中用 "Publish Point" 按钮点击多个航点
2. `click_callback` 被触发，记录航点 + 在 RViz 显示箭头和编号
3. 自动将第一个航点通过 `/goal_pose` 发送给 Nav2
4. `pose_callback` 监听 Nav2 动作状态：
   - 到达成功 → 发送下一个航点
   - 到达失败 → 重试一次，再失败则跳到下一个
   - 遍历完所有航点 → 从头开始循环

**发布的话题**：
- `/path_point` (MarkerArray) — 箭头+编号的可视化
- `/goal_pose` (PoseStamped) — 导航目标点

**订阅的话题**：
- `/clicked_point` (PointStamped) — RViz 点击
- `navigate_to_pose/_action/status` (GoalStatusArray) — 导航状态

## 5. racecar_teleop：键盘遥操作 (`racecar_teleop.py`)

**包路径**：`~/racecar/src/racecar/scripts/`

### 核心逻辑

```
键位映射:
'i'  → (linear=1, angular=0) 前进
'j'  → (linear=0, angular=1) 左转
'l'  → (linear=0, angular=-1) 右转
','  → (linear=-1, angular=0) 后退
...等
```

- 键盘读取使用 `termios` + `select`（非阻塞）
- 按下键时持续发送 Twist 消息到 `/teleop_cmd_vel`
- 速度渐变：按一次加速键增加一小步
- 松开所有键时发送 0 速度

**速度参数**：
```python
speed_start_value = 23     # 起步速度
turn_start_value = 60      # 转向最大值
speed_mid = 1500           # 中位值
turn_mid = 90              # 转向中位
```

## 6. car_controller_new：L1 路径跟踪控制器 (`car_controller_new.cpp`)

**包路径**：`~/racecar/src/racecar/src/`

### 概述

基于 L1 控制律（类似 Pure Pursuit）的路径跟踪控制器。它是导航系统的底层控制器，接收 Nav2 规划的路径 (`/plan`) 和目标点 (`/goal_pose`)，计算转向角和速度后通过 `/car_cmd_vel` 发送给底盘驱动。

### 核心类：`L1Controller` (继承 `rclcpp::Node`)

```
节点名: "art_car_controller"

订阅:
  /odom_combined (Odometry)    — EKF 融合里程计（位姿+速度）
  /plan         (Path)         — Nav2 规划的全局路径
  /goal_pose    (PoseStamped)  — 导航目标点
  /arrfinal     (Float64)      — 停止标志（0/1/2/3）
  /line_wight   (Float64)      — 车道线宽度（用于速度调节）
  /light        (String)       — 红绿灯状态 ("red"/"green")

发布:
  /car_cmd_vel  (Twist)        — 速度命令 → racecar_driver
  car_path      (Marker)       — 路径可视化标记
```

### 控制流程

```
Nav2 规划路径 → /plan ─┐
Nav2 目标点   → /goal_pose ─┤
                          ▼
               L1Controller::controlLoopCB()
                          │
              1. getEta(): 在路径上寻找前方 Lfw 距离处的目标点
                          计算车辆朝向与目标点的偏角 eta
                          │
              2. getSteeringAngle(eta): 计算转向角
                          │
              3. 应用 PD 控制：90 - steering * Angle_gain_p - Angle_gain_d * Δsteering
              4. 速度限幅 [vp_min, vp_max_base]
              5. 角度限幅 [45, 135]
                          │
                          ▼
                    /car_cmd_vel → racecar_driver
```

### 关键算法

#### 目标点选取 (`get_odom_car2WayPtVec`)

1. 将路径点从 `map` 坐标系变换到 `odom` 坐标系
2. 遍历路径点，跳过车辆后方的点
3. 找到第一个**在前方**且**距离 >= Lfw** 的路径点作为跟踪目标
4. 若车辆已接近目标点（距离 < Lfw），直接以最终目标 `odom_goal_pos` 为目标

#### 偏角计算 (`getEta`)

```cpp
// 将目标点变换到车辆坐标系
odom_car2WayPtVec.x = cos(yaw) * (forwardPt.x - carPos.x) + sin(yaw) * (forwardPt.y - carPos.y)
odom_car2WayPtVec.y = -sin(yaw) * (forwardPt.x - carPos.x) + cos(yaw) * (forwardPt.y - carPos.y)
eta = atan2(odom_car2WayPtVec.y, odom_car2WayPtVec.x)
```

#### 转向角计算 (`getSteeringAngle`)

```cpp
steeringAngle = atan2(L * sin(eta), (car2goal_dist / 2) + lfw * cos(eta)) * (180.0 / PI)
```

这是带前轮偏移量 (`lfw`) 的变体 Pure Pursuit 公式。对比标准 Pure Pursuit `δ = atan2(2 * L * sin(α), Ld)`，此公式在分母增加了 `lfw * cos(eta)` 项，用于补偿车前轮到车体中心的偏移。

### PWM 值约定

| 信号 | 最小值 | 中位（直行/停止） | 最大值 |
|------|--------|-------------------|--------|
| linear.x (速度) | 115 | 1500 | vp_max_base + 1500 |
| angular.z (转向) | 45 | 90 | 135 |

- **速度**：1500 = 停止，> 1500 = 前进，< 1500 理论上为后退（实际受驱动节点限制）
- **转向**：90 = 直行，< 90 = 左转，> 90 = 右转

最终命令值：
```cpp
cmd_vel.linear.x = baseSpeed;         // baseSpeed 是偏移 1500 后的值
cmd_vel.angular.z = 90 - getSteeringAngle(eta) * Angle_gain_p
                       - Angle_gain_d * (steering - last_steeringangle);
```

注意 `baseSpeed` 的取值是 **相对于 1500 中位的偏移量**（例如 `vp_min + 1500`），而 `angular.z` 的 90 也是中位。

### 速度控制模式

由 `/arrfinal` 话题的 `stop_flag` 值控制：

| stop_flag | 行为 | 公式 |
|-----------|------|------|
| 1.0 | 减速停车 | `baseSpeed = slow_final * baseSpeed + 1500` |
| 2.0 | 加速 | `baseSpeed = fast_final * baseSpeed + 1500` |
| 3.0 | 恢复初始速度 | `baseSpeed = initbaseSpeed + 1500` |

- `stopIdx` 确保每种模式只触发一次（暂态标志，非持续控制）
- 速度下限：`vp_min + 1500`，上限：`vp_max_base + 1500`
- 空路径时的兜底速度：115（极小爬行速度）


### 红绿灯控制

订阅 `/light`（String 类型）：
```cpp
"red"   → traffic_flag = false → 发布默认值 (1500, 90)，车辆停止
"green" → traffic_flag = true  → 正常发布控制命令
```

### 路径缓存机制 (`pathCB`)

由于路径更新频率可能低于控制周期，控制器实现了路径缓存：
1. 首次收到路径 → 保存为 `last_map_path`
2. 后续收到空路径（`poses.size() <= 0`）→ 使用 `last_map_path` 替代
3. 非空路径 → 更新 `last_map_path` 为新路径

### 关键参数

| 参数 | 默认值 | 说明 |
|------|--------|------|
| L | 0.305 | 轴距（米） |
| Lfw | 0.3 | 前视距离（米） |
| lfw | 0.1675 | 前轮到车体中心的距离（米） |
| controller_freq | 30 | 控制频率（Hz） |
| Angle_gain_p | -1.0 | 转向 P 增益 |
| Angle_gain_d | -0.0 | 转向 D 增益 |
| baseSpeed | 0.0 | 基础速度（相对 1500 的偏移） |
| baseAngle | 90.0 | 基础转向角（中位） |
| goalRadius | 0.5 | 目标到达半径（米） |
| vp_max_base | 0.0 | 最大速度偏移 |
| vp_min | 0.0 | 最小速度偏移 |
| slow_final | 1.0 | 减速系数 |
| fast_final | 1.0 | 加速系数 |
| k_rou | 0.0 | 路径曲率相关参数（当前未使用） |

### 定时器

两个定时器以不同频率运行：
- **`timer1`**（2/controller_freq ≈ 15Hz）：控制回路 `controlLoopCB`，执行路径跟踪计算和命令发布
- **`timer2`**（1/controller_freq ≈ 30Hz）：目标到达检测 `goalReachingCB`，检查距目标点是否 < goalRadius

注意 `timer1` 的频率是 `timer2` 的一半，意味着控制命令以 15Hz 发布，而目标到达检测以 30Hz 运行，确保及时停车。

### 可视化

发布 `car_path` Marker 话题（Frame: odom）：
- **points** (POINTS, 绿色) — 车辆当前位置与目标点的连线端点
- **line_strip** (LINE_STRIP, 蓝色) — 从车辆到目标点的连线
- **goal_circle** (CYLINDER, 黄色半透明) — 目标到达判定圆环

### 与导航系统的关系

```
Nav2 (AMCL + Planner + Controller)
  → /plan (全局路径) ──────────────────────┐
  → /goal_pose (导航目标) ─────────────────┤
                                           ▼
  car_controller_new (L1 路径跟踪) ──→ /car_cmd_vel
                                           │
  racecar_driver_node ─────────────────────┘
    → PWM → /dev/car (下位机)
```

控制器直接接收 Nav2 的 `/plan` 路径和 `/goal_pose`，独立完成路径跟踪（而非使用 Nav2 内置的 Controller 插件）。这意味着导航栈中的全局规划器仍在使用，但局部路径跟踪由自定义 L1 控制器替代。

## 7. 其他包简要说明

### hipnuc_imu

- ROS2 节点封装 OpenSLAM Gmapping 算法

### openslam_gmapping

- 核心 SLAM 算法库（非 ROS 节点）
- 包含 gridfastslam、scanmatcher、particlefilter 等模块
- 被 slam_gmapping 调用
